# Telegram-бот отслеживания дефекации питомца

## Цель проекта

Разработать Telegram-бота для отслеживания актов дефекации питомца.

Бот должен позволять нескольким владельцам одного питомца совместно вести журнал событий, получать уведомления о новых записях и контролировать длительное отсутствие дефекации посредством системы напоминаний.

---

# Основные сущности

## Пользователь (User)

Пользователь Telegram, взаимодействующий с ботом.

### Данные

- telegram_id
- username
- display_name

### Правила

- Один пользователь может быть привязан только к одному питомцу одновременно.
- При изменении username в Telegram он автоматически обновляется в базе данных.
- Удаление диалога с ботом или блокировка бота не влияет на данные пользователя.

---

## Питомец (Animal)

Основная сущность системы.

### Данные

- id
- name
- invite_code
- created_at

### Правила

- У питомца может быть до 20 владельцев.
- Invite-код постоянный.
- Invite-код нечувствителен к регистру.
- Формат invite-кода: 8-символьный UUID.
- При удалении последнего владельца питомец полностью удаляется из базы данных вместе со всеми связанными данными.

---

## Владение питомцем (AnimalMember)

Связь пользователя и питомца.

### Данные

- animal_id
- user_id
- joined_at

### Правила

- Все владельцы имеют одинаковые права.
- Роли отсутствуют.
- При привязке пользователя к новому питомцу он автоматически отвязывается от предыдущего.

---

## Событие дефекации (DefecationEvent)

### Данные

- id
- animal_id
- created_by_user_id
- created_at
- type
- location

### Типы событий

#### Успешный сценарий

```text
toilet
```

#### Неуспешный сценарий

```text
accident
```

### Возможные значения location

```text
toilet
sofa
bed
carpet
other
unknown
```

### Правила

- Время создания события всегда фиксируется автоматически.
- Создание событий задним числом не поддерживается.
- Время хранится в UTC.
- В интерфейсе отображается локальное время пользователя.

# Регистрация пользователя

При первом запуске отображаются кнопки:

```text
Создать питомца
Присоединиться к питомцу
```

---

# Создание питомца

Пользователь вводит имя питомца.

После создания бот показывает:

```text
Питомец создан.

Код приглашения:
XXXXXXXX
```

---

# Присоединение к питомцу

Пользователь вводит пригласительный код.

Если код найден:

```text
Вы подключены к питомцу <имя>.
```

Если код не найден:

```text
Код не найден.
Введите код повторно.
```

Если достигнут лимит владельцев:

```text
Невозможно присоединиться.
Достигнут лимит владельцев.
```

После присоединения новому владельцу показываются 5 последних событий питомца.

---

# Уведомления о владельцах

При присоединении владельца всем существующим владельцам отправляется уведомление:

```text
<username> присоединился к питомцу.
```

При отвязке владельца:

```text
<username> больше не является владельцем питомца.
```

---

# Главное меню

Кнопки:

```text
✅ В туалет

❌ Не в туалет

📜 История

↩️ Удалить последнее событие

🔑 Показать код приглашения

❓ Помощь
```

---

# Создание события

## Успешный сценарий

Пользователь нажимает:

```text
✅ В туалет
```

Создается событие:

```text
type = toilet
location = toilet
```

Пользователю отображается:

```text
Событие создано.
```

Остальным владельцам:

```text
<username> сообщил о дефекации питомца.
```

---

## Неуспешный сценарий

Пользователь нажимает:

```text
❌ Не в туалет
```

Бот показывает:

```text
Где произошло?
```

Кнопки:

```text
Диван
Кровать
Ковёр
Другое
```

После выбора создается событие:

```text
type = accident
location = выбранное значение
```

Автору:

```text
Событие создано.
```

Остальным владельцам:

```text
<username> сообщил о дефекации питомца вне туалета.
```

---

# Незавершенные сценарии

У пользователя одновременно может существовать только один незавершенный сценарий.

Если пользователь каким-либо образом запускает новый сценарий:

- новый сценарий становится активным;
- старый незавершенный сценарий удаляется.

Если пользователь не завершил сценарий в течение одного часа:

автоматически создается событие:

```text
type = accident
location = unknown
```

Если после автоматического завершения пользователь нажимает кнопку из старого сообщения:

```text
Этот сценарий уже завершен.
```

---

# Ограничение частоты событий

Для каждого пользователя действует ограничение:

```text
не более одного события в минуту
```

При нарушении ограничения:

```text
Невозможно добавить событие.
Попробуйте позже.
```

---

# История событий

Отображается от новых записей к старым.

Формат:

```text
1. ✅ В туалет
17.06.2026 14:32
@username

2. ❌ Диван
17.06.2026 12:11
@username
```

Формат времени:

```text
DD.MM.YYYY HH:MM
```

Пагинация:

```text
5 записей на страницу
```

Если история пуста:

```text
История пуста.
```

---

# Удаление последнего события

Пользователь нажимает:

```text
Удалить последнее событие
```

Бот показывает подтверждение:

```text
Удалить последнее событие?
```

Кнопки:

```text
Да
Нет
```

После удаления отображаются оставшиеся записи.

Удаляется только последнее событие.

Удаление возможно без ограничения по времени.

Удаление полностью физическое.

Удаленные записи не отображаются в истории.

Если история пуста:

```text
Нет событий для удаления.
```

Если два пользователя одновременно удаляют последнее событие, допускается удаление двух последних событий.

---

# Напоминания

Напоминания считаются на уровне питомца.

Отсчет ведется от времени последнего события питомца.

При появлении нового события счетчик полностью сбрасывается.

## Первое напоминание

Через:

```text
48 часов
```

после последнего события.

## Повторные напоминания

Через каждые:

```text
8 часов
```

после первого напоминания.

## Получатели

Напоминание получают все владельцы питомца.

## Текст напоминания

```text
Давно не было записей о дефекации питомца <имя>.
```

Если событий нет:

```text
напоминания не отправляются
```

Если последнее событие удалено:

- пересчитывается новое последнее событие;
- цикл напоминаний пересчитывается немедленно.

---

# Event Log

Для каждого питомца ведется отдельный журнал событий.

Лог доступен только через специальную команду.

В основном интерфейсе не отображается.

Лог хранится бессрочно.

Пагинация:

```text
5 записей на страницу
```

## Типы записей

```text
EVENT_CREATED
EVENT_DELETED
NOTIFICATION_SENT
USER_JOINED
USER_LEFT
ERROR
```

## Логирование уведомлений

Логируются:

- успешные отправки;
- неуспешные отправки.

При ошибке доставки создается запись:

```text
ERROR
```

---

# Работа с Telegram

Удаление диалога с ботом не влияет на данные.

Блокировка бота не влияет на данные.

Пользователь остается владельцем питомца даже после блокировки бота.

Если все владельцы заблокировали бота:

- питомец сохраняется;
- история сохраняется;
- лог сохраняется.

---

# Отображение пользователей

В событиях, уведомлениях и истории отображается:

```text
@username
```

В базе данных пользователь идентифицируется по:

```text
telegram_id
```

В событиях хранится:

```text
created_by_user_id
```

Имя пользователя определяется через таблицу владельцев.

# Требования к принятию архитектурных решений агентом

Если в ТЗ отсутствует описание конкретного сценария, агент должен руководствоваться следующими принципами:

1. Простота реализации имеет приоритет над расширяемостью.
2. Решение не должно усложнять пользовательский интерфейс без необходимости.
3. Решение не должно изменять уже зафиксированную бизнес-логику.
4. Решение не должно приводить к потере пользовательских данных, кроме случаев явно описанного удаления питомца.
5. При конфликте между удобством реализации и консистентностью данных приоритет отдается консистентности данных.
6. Все даты и время должны храниться в UTC.
7. Все операции изменения данных должны логироваться в Event Log.
8. Любая ошибка, влияющая на корректность работы системы, должна логироваться как ERROR.
9. Все ограничения должны проверяться на стороне сервера независимо от ограничений пользовательского интерфейса.
10. При возникновении неоднозначности следует выбирать поведение, минимизирующее вероятность появления ложных записей о дефекации.
