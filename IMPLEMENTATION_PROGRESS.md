| Компонент | Решение |
|-----------|---------|

| Требование | Статус |
|------------|--------|
| Один активный сценарий на пользователя | ✅ (`pending_scenarios.user_id` UNIQUE) |

| Таймаут 1 ч → `accident` + `unknown` | ✅ (фоновый worker, проверка каждые 60 с) |
| «Этот сценарий уже завершен.» на устаревшую кнопку | ✅ |
| Хранение в БД | ✅ (таблица `pending_scenarios`) |
| Сброс при смене питомца | ✅ (`leave_current_animal`) |

**Файлы:** `src/db/models.py` → `PendingScenario`; `src/services/scenarios.py`; `src/bot/handlers/events.py`; миграция `002_pending_scenarios`

**Решение:** таймаут обрабатывается asyncio-worker при старте бота (без APScheduler — он запланирован на этап 5 для напоминаний). При авто-завершении уведомляются остальные владельцы (как при обычном accident); автору отдельное сообщение не отправляется (в ТЗ не описано).

#### Детали реализации (этап 3)

**Новая таблица `pending_scenarios`** (миграция `002_pending_scenarios`):

| Поле | Назначение |
|------|------------|
| `id` | Идентификатор сценария (передаётся в `callback_data` inline-кнопок) |
| `user_id` | UNIQUE — не более одного активного сценария на пользователя |
| `animal_id` | Питомец, для которого начат сценарий |
| `started_at` | UTC-время старта; по нему считается таймаут |

**Сервис `src/services/scenarios.py`:**

| Функция | Поведение |
|---------|-----------|
| `start_accident_scenario` | Удаляет предыдущий сценарий пользователя, создаёт новый |
| `cancel_user_scenario` | Удаляет активный сценарий без создания события |
| `get_user_scenario` | Ищет сценарий по `id` и `user_id` (защита от чужих callback) |
| `complete_scenario` | Удаляет сценарий после успешного выбора location |
| `process_expired_scenarios` | Для просроченных: `accident` + `unknown`, audit log, уведомления |
| `run_timeout_worker` | Фоновый цикл с отдельной сессией БД и `commit` |

**Константы таймаута** (в `scenarios.py`, не в `.env`):

- `SCENARIO_TIMEOUT_SECONDS = 3600` — 1 час до авто-завершения
- `TIMEOUT_CHECK_INTERVAL_SECONDS = 60` — интервал проверки просроченных сценариев

**Изменения в хендлерах (`events.py`):**

- «❌ Не в туалет» → `start_accident_scenario` + клавиатура `acc_loc:{scenario_id}:{location}`
- «✅ В туалет» → `cancel_user_scenario`, затем создание события `toilet`
- Callback выбора места → проверка `scenario_id`; если сценарий не найден — `SCENARIO_ALREADY_COMPLETED`
- При rate limit на callback сценарий остаётся активным (можно повторить позже)

**Прочие изменения:**

- `src/bot/texts.py` — константа `SCENARIO_ALREADY_COMPLETED`
- `src/bot/keyboards.py` — `accident_location_keyboard(scenario_id=...)`
- `src/main.py` — `asyncio.create_task(run_timeout_worker(bot))` при старте, отмена при остановке
- `src/services/animals.py` — `cancel_user_scenario` в `leave_current_animal` (смена/создание питомца)

**Поток незавершённого сценария:**

```text
«Не в туалет» → запись в pending_scenarios → inline «Где произошло?»
  ├─ выбор места → событие accident → удаление сценария
  ├─ «В туалет» / новый «Не в туалет» / смена питомца → удаление без события
  ├─ устаревшая кнопка → «Этот сценарий уже завершен.»
  └─ таймаут 1 ч → accident + unknown → удаление сценария → уведомление владельцам
```

## Как запустить локально

```bash
cd pet-poop-bot
cp .env.example .env          # вставить BOT_TOKEN от @BotFather
docker compose up -d          # PostgreSQL
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python -m src.main
```

| Переменная | Описание |
|------------|----------|
| `BOT_TOKEN` | Токен Telegram-бота |
